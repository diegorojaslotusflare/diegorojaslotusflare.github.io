Periscope Metrics Management

This API is used to provide data for the Periscope product that should allow our clients to view and easily understand all supported growth metrics. Metrics can then be further refined by filtering them by segments or by creating cohorts and using them instead or along side segments.

Each metric has its time step (period of computation). If for example, a metric in the range from 2019.01.01. to 2019.03.01. is requested, and it is a monthly metric the date range will be split into one month long 'buckets'. In this example, the resulting buckets would be 2019.01.01 and 2019.02.01. It is possible to request a daily and a monthly metric together, the results will include all the buckets for both metrics as if they were called separately.

If a metric is considered aperiodic as L28 is at the time of writing, that would mean that l28 will be computed with regards only to time_till as the relevant date.

For now available metrics sorted by time steps are:

  • Daily

    • dau - Daily Active Users
    • total_users - Originaly Year To Date, but as the time range is provided in the request it is now total_users. Represents the total number of unique users in the given time range

  • Weekly

    • wau - Weekly Active Users
    • 7_day_ret - 7-day retention rate

  • Monthly

    • mau - Monthly Active Users
    • mau_ret - The number of Monthly **Active Users that are returning users
    • mau_new - The number of Monthly **Active Users that are new users
    • mau_res - The number of Monthly **Active Users that are resurrected users
    • mau_ret_proc - The procentage of all Monthly **Active Users that are returning
    • mau_new_proc - The procentage of all Monthly **Active Users that are new
    • mau_res_proc - The procentage of all Monthly **Active Users that are resurrected
    • retention_rate - Total user monthly retention rate
    • retention_rate_new - Retention rate of new users
    • retention_rate_ret - Retention rate of returning users
    • retention_rate_res - Retention rate of resurrected users
    • churn_rate - Total user monthly churn rate (1 - retention_rate)
    • churn_rate_new - Churn rate of new users (1 - retention_rate_new)
    • churn_rate_ret - Churn rate of returning users (1 - retention_rate_ret)
    • churn_rate_res - Churn rate of resurrected users (1 - retention_rate_res)
    • arpu - Average Revenue Per User
    • ltv - Lifetime Value
    • porting_in - The number of users that have ported-in from another operator within a time period.
    • porting_out - The number of users that have ported-out from another operator within a time period.
    • trial_to_paid - The percentage of users who have converted into paid customers after joining the trial within a time period.

  • Aperiodic

    • new_ret - Retention of new users cohorts
    • aha_ret - Aha moment - Frequency (Retention rate for users who were active for 1,2,…,N days on their first N days since they joined)
    • aha_ret_long - Aha moment - Frequency vs. Engagement (2 month retention after 3 months passed for users who were active N days on their first N days since joined)
    • l28 - how many users have been active for every N number of days, where N goes from 1 to 28

**Metrics Computation (/api/v3/growth/metric)**

is a service that computes metrics for segment in the time range [time_from; time_till).

It is also possible to provide one metric and multiple segments and get the results grouped by segment in the time range [time_from; time_till)

Results are cached in Redis. If the metric was already computed, it will not be recomputed again and previous results will be fetched from cache.

Important notes about time ranges:

  • time_from is inclusive and time_till is exclusive.

  • If the time_from parameter requires data from before precomputations were introduced, the response will return the results for which the data is present. If there is no data present for the requested time range the response will return with the code 12 - NO_DATAPOINTS_COMPUTED

  • If the time_till parameter is in the future it will be rounded down to today.

  • If time_till refers to today, but an hour hasn't passed since midnight it will be rounded down to yesterday.

  • time_format, an optional parameter, defaults to epoch. This means the endpoint will only accept time_from and time_till as epoch timestamps.

  • If time_format is passed with the value human the endpoint will expect time_from and time_till to be stringified dates in format yyyy-mm-dd. Also if the buckets returned have timeEpoch as a field, they will have an additional field timeHuman next to it.

  • L28 will always return only the data for 28 days with the final date being time_till

  • Monthly metrics should always have both time_from and time_till set to the first day of the month. (if not, both dates will be floored to the first day of the respected month before the processing begins)

Other important notes

  • If more than one metric is passed and ONLY one segment, the response will have bundled metric results as if it was called with each metric separately for that segment.

  • Same as above if more than one segment is passed to the endpoint and ONLY one metric, the response will have bundled metric results as if it was called with each segment separately.

  • As with most other routes in our system, the operator name is NOT passed inside the body, but is a part of the JWT. This is the token that is used for the authentication in most of our endpoints including this one.

  • The "Default Dashboard" refers to results that have not been combined with any segments. To get these results the segments list should have an element with the value "default".

  • The endpoint can return a maximum of 500 data points in one response across all metrics requested. For example means it can compute DAUs for 500 days.

  • Each bucket will have an ID, that is used when calling a different endpoint for cohort creation. When called with multiple segments the ID will restart its calculation for every segment.


**Cohort Creation (api/v3/growth/cohort/create)**

Similarly to the endpoint described in above for /api/v3/growth/metric. This endpoint also computes a metric for a segment in the time range [time_from; time_till) but uses the provided data to create a new segment-like structure called a cohort.

The big differences between this and the metric computation endpoint is that this endpoint accepts ONLY one metric, ONLY one segment as well as a list of data_points and a cohort name.

data_points is a list passed in the request that specifies which buckets should be used to construct a cohort

For example if a request was sent to /api/v3/growth/metric to compute DAU for range [2019-01-01, 2019-01-04) it's response would include 3 buckets representing the 3 days of active users. Each of these buckets would have an id field. Now if we want to create a new cohort that can be used to get metrics only for users that were active on 2019-01-01 and 2019-01-03 we would call this endpoint with a list of data_points who's elements equal to ID-s of desired buckets.


**All notes that were specified for the endpoint /metric also apply here.**

All cohorts are stored in Redis. They are represented as a bitmap and as a meta table. The meta table has the following information about the cohort:

  • bitmap_id
  • name
  • creation_time

So for each new cohort in total 4 keys are created.

Not all metrics are supported. Here is the list of the ones that are:

  • Daily

    • dau

  • Weekly

    • wau

  • Monthly

    • mau_ret
    • mau_new
    • mau_res
    • mau

  • Aperiodic

    • l28

There are also other cohort operations described here such as: - **Cohort deletion** (api/v3/growth/cohort/delete) - **Cohort listing** (api/v3/growth/cohort/get) - **Cohort renaming** (api/v3/growth/cohort/rename)

**Debug Endpoints**

These are:

  • /api/v3/growth/debug/clear_cache
  • /api/v3/growth/bitmaps_exist

These endpoints are designed ONLY for debugging purposes. Once clear_cache endpoint is called it will remove all cached data created by calling other periscope endpoints (this does NOT include cohorts)

When bitmaps_exist endpoint is called with a metric and a time range, it will check if there are precomputed bitmaps for that metrics in the provided range and return for which dates the maps exist and for which dates they don't.

Here metric represents one of the metrics that Octopus precomputes, not the ones that this API calculates. Here is a list of them (only dau is daily, the rest are monthly):

  • dau
  • mau
  • pau
  • mau_mew
  • mau_ret
  • mau_res
  • revenue
  • doasj_N

post /cohort/create
(cohortCreatePost)
Given the datapoints representing bucket id-s from the metric computation request with the rest of the parametars (time range, metric, segment) it creates a cohort for those datapoints. A cohort is a static segment representing the users that were included in the provided datapoints. For example if we request to make a cohort from DAU in range from 2019-01-01 to 2019-01-05 days, with datapoints "0" and "3" the resulting cohort will represent users that were active on 2019-01-01 and 2019-01-04.
