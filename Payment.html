post /check_payment_validity
Validate if default payment is valid [TENANT only] (checkPaymentValidityPost)
In order to check if customer's default payment method is valid reservation of 1$ will be tried and if everything is okay release of reserved amount to customers account will be done. In case default payment method is not valid (validation/verification failed) or customer doesn't have enough money - error will be returned - and result will be saved in recurrence_event for passed order_id.

Managing (adding/deleting/updating) payment methods (managePaymentMethodsPost)

Payment_methods array should contain actions that should be performed on specific payment method. Since array is provided, actions will be executed in provided order. Payment data should contain payment_id (token/nonce) and action that should be performed (with optional additional_properties)

  • payment_id - in Braintree for INSERT action payment_id should be only payment NONCE, and for UPDATE/DELETE should be payment TOKEN.
  • update of apple pay payment method - for payment method updating action INSERT + payment_id NONCE are always used, since apple pay can't be updated (so new updated one needs to be created and old one should be removed)

post /refund_or_cancel
Refunds/Voids the transaction for the specified amount [TENANT only] (refundOrCancelPost)

Refunds/Voids user's transaction for the specified amount. There are two payment return actions - REFUND and VOID.

VOID case:

In case external braintree transaction is in one of ["authorized", "submitted_for_settlment"] states that means funds were not removed yet from user's bank account (they were reserved), so that transaction will be voided. When voiding the transaction, FULL (original) amount will be returned to the user.

REFUND case:

In case external braintree transaction is in one of ["settled", "settlement_pending", "settling] states thar means funds were removed from user's bank account, so that transactions will be refunded. Partial perfunds are allowed.
